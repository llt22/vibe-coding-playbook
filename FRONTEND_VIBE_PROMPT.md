# 前端 Vibe 工作流

> 你不是要变成 UI 架构师，而是要建立一套机制，让 AI 产出的前端代码**风格统一、交互完整、状态可控、可安全迭代**。

---

## 1) 三条铁律（前端版）

**铁律 A：先契约再页面**
- 接口返回的数据结构确定了吗？哪些字段可能为空？错误码有哪些？
- 类型定义（TypeScript/Zod/OpenAPI 生成）是否先于 UI 代码存在？
- **没有稳定的数据契约就开始写页面，等于在沙子上盖楼**

**铁律 B：先可观测再可调试**
- 页面报错时，能从监控系统定位到：哪个用户、哪个页面、哪个接口、什么错误吗？
- 关键用户操作（注册/支付/核心流程）有埋点吗？能知道完成率和流失点吗？
- 不是靠用户截图反馈 bug，而是系统主动告诉你哪里出了问题

**铁律 C：先可回滚再上新**
- 这次改动出问题能快速关掉吗（feature flag / 灰度开关）？
- 构建是否可复现？依赖版本是否锁定？
- "能跑"不算完成，"出问题能 30 秒内关掉"才算完成

---

## 2) 责任分界线

### 你必须定义（AI 猜不到的）
1) **设计系统**：用什么组件库、design token 长什么样、现有组件有哪些
2) **交互规格**：用户操作流程、状态流转、跳转逻辑、权限控制
3) **数据契约**：接口返回什么结构、哪些字段可能为空、分页策略
4) **边界场景的产品决策**：断网怎么表现？加载超时怎么办？空数据显示什么？
5) **性能预算**：首屏可接受的加载时间？哪些路由需要懒加载？新依赖的体积上限？

### AI 可以交付（你验收结果）
- 基于设计稿/截图还原组件
- 响应式适配、状态管理样板、测试用例骨架
- 重复性 CRUD 页面（列表/详情/编辑）
- 基于你的规范铺开埋点和错误处理

**你定义"长什么样、怎么交互、边界怎么处理"，AI 负责写出来并自证正确。**

---

## 3) 质量门（DoD）+ 风险分级

按**页面/组件的风险等级**分档，不用每个组件全量达标：

**F0 静态展示**（纯展示页面/组件）
- 类型通过、lint 通过
- 使用项目 design token / 组件库
- props 接口清晰（TypeScript 类型定义，必填/可选/默认值明确）

**F1 数据读取**（列表/详情/搜索/筛选）= F0 +
- 五态完备：**loading / error / empty / 正常**（缺哪个补哪个）
- 接口字段为空/异常时不白屏（兜底渲染）
- 请求走统一封装（带 trace_id、统一错误处理、超时控制）

**F2 数据写入**（表单/编辑/创建/删除）= F1 +
- 表单校验 + 错误提示 + 防重复提交
- 写操作有并发保护（按钮禁用/节流/幂等键/取消旧请求）
- 权限 gating（无权限的入口隐藏或禁用，不只是后端拦截）
- 测试：至少覆盖 成功 / 失败 / 无权限

**F3 关键链路**（支付/登录/核心业务流程/AI 调用）= F2 +
- feature flag 或灰度开关（出问题能快速关掉）
- E2E 测试覆盖完整流程
- 性能预算检查（bundle 变化、路由性能）
- 可访问性检查通过

---

## 4) 组件结构要求

AI 最爱把所有逻辑塞进一个组件文件里。你要问的是：

- 这个组件能**不依赖接口/全局状态**独立渲染吗？（如果不能，展示和逻辑没分开）
- 数据获取是在**靠近页面顶层**集中发起，还是在子组件深处各自偷偷调？（后者 = 瀑布流请求）
- 单个文件**超过 300 行**了吗？（超过就该拆）
- 组件的样式会**影响外部**吗？（CSS 泄漏 = 维护噩梦）

目标结构：
- **展示组件**：只接收 props，只负责渲染，可独立预览
- **逻辑 Hook**：数据获取、状态管理、副作用，可独立测试
- **页面/容器组件**：组装展示组件和 hook，处理路由和权限

---

## 5) 状态管理：先问"这个状态属于谁"

AI 特别容易制造"状态灾难"——同一份数据在三个地方各存一份。

每次遇到状态管理问题，先分类：
- 这是**服务端返回的数据**吗？→ 用数据缓存库管理，不要手动存 state
- 这是**全局 UI 状态**吗（主题/语言/侧边栏）？→ 全局 store
- 这是**局部 UI 状态**吗（弹窗/hover/输入）？→ 组件内 state，不要上提到全局
- 这个状态**能从 URL 表达**吗（筛选/分页/tab）？→ 放 URL params，别放 state

**服务端数据用缓存库，UI 状态就近管理，URL 能表达的别用 state。**

---

## 6) 请求层与安全

### 请求层
- 所有接口请求是否走**统一封装**？（不散写 fetch/axios）
- 统一封装是否自动处理：认证信息注入、trace_id、超时、错误归一化、401/403 跳转？
- 涉及并发的场景（搜索/筛选/分页）：**后发请求会覆盖先发结果吗**？还是会出现竞态闪烁？

### 安全（前端实打实会踩的）
- 用户 token 存储策略安全吗？（存哪里？刷新机制？）
- 有没有直接渲染用户输入的 HTML？（XSS 入口）
- 新引入的第三方依赖是否可信？版本是否锁定？

---

## 7) 前端可观测

**错误监控**
- 全局错误有统一捕获吗？（接口错误 + JS 运行时错误 + Promise 未处理拒绝）
- 错误上报包含：用户标识、页面路径、接口 URL、错误信息、环境信息吗？
- 关键流程有埋点吗？能看到**完成率和流失点**吗？

**性能指标**
- 核心页面的 LCP / CLS / INP 有在跟踪吗？
- Bundle 大小有监控吗？超预算会报警吗？

**问题定位**
- 能从监控系统还原用户操作路径吗？还是只能靠用户描述复现？

---

## 8) 性能问题定位协议

页面变慢时：

1) **先量化**：到底慢多少？LCP / 首屏加载 / 交互响应 具体数字是什么？
2) **分类定位**：
   - **加载慢** → bundle 哪个包太大？有没有按需加载？
   - **渲染慢** → 哪个组件重渲染太多？
   - **交互卡** → 主线程有没有被长任务阻塞？
3) **AI 常见性能坑**：
   - 不必要的重渲染（缺少稳定引用 / memo）
   - 首屏加载全部路由（没有懒加载）
   - 大列表全量 DOM 渲染（没有虚拟化）
   - 子组件各自请求接口（瀑布流）
4) **修复顺序**：先拆包/懒加载 → 再减少重渲染 → 再虚拟化 → 最后才考虑 SSR

---

## 9) Prompt 包

别说"写个用户列表页"，而是给 AI 这些输入：

1) **设计参考**：截图/设计稿/相似页面
2) **组件库与 token**：用什么 UI 库、间距/颜色/字号规范
3) **数据契约**：接口返回结构、字段类型、哪些可能为空
4) **交互规格**：操作流程、状态流转、跳转逻辑
5) **五态要求**：loading / error / empty 分别什么样
6) **边界场景**：超长文本、极大数量、无权限、离线

简单展示组件（F0/F1）给 1-3 就够。复杂交互页面（F2/F3）建议完整输入。

## 10) 审计 Prompt

```
按以下清单检查本次前端改动：
1. 契约：类型定义是否先于 UI 存在？是否和接口一致？有无 any？
2. 设计一致性：是否使用项目 design token / 组件库？有无硬编码样式值？
3. 五态完备：loading / error / empty / 正常 是否都处理了？
4. 数据健壮性：接口字段为空时是否兜底？列表为空有无空状态？
5. 请求规范：是否走统一封装？并发场景有无竞态保护？
6. 安全：有无直接渲染用户输入 HTML？token 存储策略是否安全？
7. 组件结构：展示与逻辑是否分离？单文件是否超 300 行？数据获取是否集中？
8. 状态管理：服务端数据是否用缓存库？有无重复存储同一份数据？
9. 可访问性：语义化标签？键盘可操作？alt/label 齐全？
10. 性能：有无不必要重渲染？大列表是否虚拟化？路由是否懒加载？
输出表格：检查项 | 通过/不通过 | 证据 | 风险说明 | 修复建议。
不通过项直接给修复 diff。
```

## 11) 存量项目渐进改造顺序

先止血、再统一、再补全（不要一上来重构所有组件）：

1. **统一请求层 + 全局错误捕获**：立刻让"白屏崩溃"变成"有兜底的错误页"
2. **提取 design token + 统一组件库**：消灭"自己写的按钮"，新页面开始统一
3. **拆分最大的几个巨型组件**：展示 + hook + 容器分离
4. **逐页补全五态**：用审计 Prompt 批量让 AI 扫描缺失的 loading/error/empty
5. **性能治理**：bundle 分析 → 路由懒加载 → 图片优化 → 大列表虚拟化

> 关键技巧：新代码按规范写，旧代码按优先级逐步迁移——不要试图一次性重写。
