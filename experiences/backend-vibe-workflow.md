# 后端 Vibe 工作流

> 你不是要变成全栈老后端，而是要建立一套机制，让 AI 产出**可控、可验证、可回滚**。

---

## 1) 三条铁律（对抗 AI 的"乐观主义"）

**铁律 A：先写不变量（Invariants），再写实现**
- 这条接口/流程**允许重复请求吗**？重复 5 次结果是否一致（幂等）
- 写入是否需要**原子性**（事务边界）
- 数据是否必须满足**唯一性/引用完整性/状态机**（DB 约束/状态流转）

**铁律 B：先可观测（Observability），再谈可调试**
- 没有 request_id/结构化日志/耗时打点，你就无法"vibe 上线"
- 目标不是能打断点，而是能从日志/指标定位：慢在 DB？慢在外部调用？慢在锁？

**铁律 C：先可重复交付（Repeatability），再追求架构美**
- migration 可重复、配置分环境、发布可回滚
- "能跑"不算完成，"能安全迭代"才算完成

---

## 2) 责任分界线

### 必须你掌握（AI 只能辅助，不能代替你拍板）
1) **数据建模与约束**：表关系、unique、索引、事务边界
2) **授权（AuthZ）规则**：谁能操作哪条数据（BOLA/IDOR 防线）
3) **幂等与并发策略**：去重键、状态机、重试与回滚策略
4) **上线与迁移兼容**：先加字段/后读写、双写/回填、回滚策略
5) **可观测性标准**：要打哪些日志字段、哪些指标、慢查询怎么抓

### 可以主要交给 AI（你做 Review + 测试兜底）
- CRUD/路由/DTO/中间件样板代码
- OpenAPI/Swagger 初稿、请求响应示例
- 基于你给定 schema 的 ORM 查询与 migration 初稿
- 测试用例骨架、Mock/Fixture 生成
- 日志埋点"按你的规范"铺开

**AI 负责"写"，你负责"定义不变量 + 验证 + 上线安全"。**

---

## 3) Definition of Done（DoD）

每 vibe 一个后端功能，满足下面 6 条才允许合并/上线（否则就是债）：

1) **契约**：OpenAPI/接口文档 + 错误码表（至少成功/参数错/未授权/越权/冲突/限流）
2) **校验**：入口 schema validation（缺字段/类型错/非法值）
3) **授权**：资源级校验（user 是否 owner / 是否有 role）
4) **幂等/事务**：写操作声明是否幂等；需要则有幂等键 + DB unique；事务边界明确
5) **可观测**：request_id、关键字段结构化日志、DB/外部调用耗时
6) **测试**：至少 3 个集成测试：成功、越权、重复请求/并发（或冲突）

按**风险等级**分档执行，不用每个接口全量达标：
- **R0 只读**（列表/详情）：DoD 第 1、2、5 条
- **R1 一般写入**（创建/编辑）：DoD 第 1-3、5 条 + 至少 2 个测试
- **R2/R3 关键写入/外部副作用**（扣费/LLM调用/支付）：DoD 全部 6 条

---

## 4) 可测结构要求

AI 最爱把逻辑糊在 controller 里，导致难测。你要的结构是：

- **Domain（纯函数）**：业务规则/状态机（不直接读写 DB，不直接发 HTTP）
- **Use case / Service（编排）**：事务、调用 repository、调用外部服务（可注入依赖）
- **Adapters（IO）**：DB repository、HTTP client、queue publisher

三条硬性要求：
1) **依赖注入**：db、clock、uuid、httpClient 都从参数传入（测试可替换）
2) **纯逻辑可单测**：核心规则函数不访问环境、不读 process.env
3) **集成测试可跑**：提供 testcontainers/docker-compose（或至少可切换到测试库）

## 5) LLM 集成场景的不变量

当后端涉及 LLM 调用/流式输出时，除了铁律 A-C，还要回答这些问题：

**成本与额度**
- 这次调用**花多少钱**？用户有额度限制吗？超额怎么处理？
- 如果请求重试，会不会**重复扣费**？
- 哪些错误可以安全重试（429/503），哪些不行（400/已消费 token）？

**流式输出的"完成"定义**
- 什么算**"这次调用成功了"**？流到一半断了算成功还是失败？
- 客户端断开时，**上游 LLM 调用是否终止**（还是空跑浪费 token）？
- 部分生成的结果要**保存还是丢弃**？

**幂等与续传**
- 同一个问题重复提问，**返回缓存还是重新生成**？
- 前端重连时，如何识别"是同一次请求"并恢复上下文？

**外部依赖容错**
- LLM 超时通常 10s-120s，和普通 API 的 3-5s 完全不同——**超时策略是否匹配**？
- LLM 不可用时**降级方案**是什么？（排队/错误码/缓存上次结果）
- 知识库/搜索等辅助服务失败时，**是否阻塞主流程**？

**可观测**
- 每次 LLM 调用是否记录：model、input/output tokens、latency、status？
- 是否能按用户/功能/模型维度**归因成本**？

## 6) 慢 SQL 定位协议

1) **先看指标**：哪个 endpoint P95/P99 飙升？错误率是否上升？
2) **再看日志**（按 request_id）：该请求总耗时里，DB 占比多少？外部调用占比多少？
3) **抓慢查询**：开启慢查询日志/统计，定位 Top SQL
4) **三大常见元凶**（AI 特别容易写出来）：
   - N+1：循环里查 DB
   - 缺索引：where/join/order by 字段没索引
   - 错误分页：offset 很大导致越翻越慢（考虑 cursor）
5) **修复顺序**：先加索引/改查询形状 → 再考虑缓存 → 最后才是架构改造

## 7) 常用指令包

别说"写个下单接口"，而是一次性给 AI 这 7 项输入：

1) DB schema（含约束/索引）
2) API 契约（请求/响应/错误码）
3) 授权规则（资源归属/角色）
4) 幂等要求（是否需要 Idempotency-Key、unique 键设计）
5) 事务边界（哪些操作必须同一事务）
6) 可观测标准（日志字段 + 耗时打点）
7) 测试清单（成功/越权/重复请求/并发）

简单功能（R0/R1）不用 7 项全给，只需要给差异项。复杂功能（R2/R3）建议完整输入。

---

## 8) 审计 Prompt（代码生成后让 AI 自检）

```
你现在是审计员。按以下清单逐条检查本次改动：
1. 契约：接口文档是否更新？错误码是否完整？
2. 校验：入参是否有 schema 校验？
3. 授权：资源操作是否校验归属？
4. 幂等/事务：写操作是否幂等？事务边界是否明确？
5. 可观测：是否有 request_id、结构化日志、耗时打点？
6. 测试：是否覆盖成功、越权、重复请求？
输出表格：检查项 | 通过/不通过 | 证据 | 风险说明 | 修复建议。
不通过项直接给修复 diff。
```

## 9) 存量项目渐进改造顺序

先止血、再加固、最后重构（不要一上来全量 DoD）：

1. **L1 可观测止血**：统一 error handler + request_id + 关键耗时（1-2 天见效）
2. **L3 授权补齐**：先补越权（最容易被打穿）
3. **L4 数据正确性**：再补幂等/事务/DB 约束（最容易产生"慢性数据事故"）
4. 最后才是分层重构/架构美化

> 关键技巧：用"绞杀者模式"——先用中间件在外层加护栏，不急着把内部全部重构。
